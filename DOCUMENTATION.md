# XT POS — Technical Documentation

Developer-facing reference for the XT POS application: how it is structured,
how the pieces fit together at runtime, the data model, the HTTP surface, and
the build / install / update / uninstall pipeline.

For end-user install instructions and a feature overview see [README.md](README.md).
For release history see [CHANGELOG.md](CHANGELOG.md).

---

## 1. What it is

A single-machine Windows point-of-sale application. A Flask app is served
locally by [waitress](https://pypi.org/project/waitress/) and rendered inside a
native desktop window (Microsoft Edge **WebView2**, via
[pywebview](https://pywebview.flowrey.org/)) — there is no browser and nothing
goes over the network. All data lives in a local **MariaDB** database accessed
through SQLAlchemy + PyMySQL.

The whole thing compiles to `POS.exe` (PyInstaller) and ships as a single
self-contained installer, `XTPOS-Setup.exe`, that downloads and installs
MariaDB, creates the database, and lays down Start-Menu / Desktop shortcuts.

| Concern        | Choice |
|----------------|--------|
| Language       | Python 3.10+ |
| Web framework  | Flask + Jinja2 templates + vanilla JS |
| WSGI server    | waitress (local, background thread) |
| Desktop shell  | pywebview (embedded WebView2) |
| ORM / driver   | SQLAlchemy + PyMySQL |
| Database       | MariaDB (local service) |
| Packaging      | PyInstaller (`*.spec`) |
| Installer UI   | tkinter wizard (`installer_app/`) |

## 2. Runtime architecture

```
                          POS.exe  (one process)
   ┌──────────────────────────────────────────────────────────────┐
   │  launcher.serve()                                              │
   │                                                                │
   │   1. provision.bootstrap()  ── ensures MariaDB + pos_db are    │
   │                                reachable (may hand off to the  │
   │                                installer and exit)             │
   │                                                                │
   │   2. background thread: waitress serves the Flask app on       │
   │      127.0.0.1:<port>                                          │
   │                                                                │
   │   3. main thread: pywebview opens a native window pointed at   │
   │      http://127.0.0.1:<port>   (falls back to the browser if   │
   │      no WebView2 runtime is present)                           │
   └──────────────────────────────────────────────────────────────┘
                                  │  SQLAlchemy / PyMySQL
                                  ▼
                       MariaDB service  →  pos_db
```

Key points:

- **Single process, two threads.** `launcher.serve()` ([launcher.py](launcher.py))
  runs the WSGI server on a daemon thread and owns the GUI window on the main
  thread. Closing the window ends the process and stops the server.
- **Port selection.** `_pick_port()` uses the preferred port (`POS_PORT`,
  default 5000) if free, otherwise an OS-assigned free port — so a stray
  `python app.py` squatting on 5000 never causes the window to show the wrong
  page.
- **Readiness gate.** `_wait_until_up()` polls the local URL before opening the
  window so the user never sees a connection error on launch.
- **Graceful DB failure.** If MariaDB is unreachable, SQLAlchemy raises
  `OperationalError` / `InterfaceError`, which a global error handler turns into
  a friendly `db_error.html` page (HTTP 503) instead of a stack trace.

## 4. `launcher.py` — the three modes

| Invocation | Behaviour |
|------------|-----------|
| `POS.exe` | Run `serve()`: self-heal the DB, start waitress, open the native window. |
| `POS.exe --init-db --config <file>` | `init_db_from_config()`: connect as `root` with the password the MSI just set, `CREATE DATABASE`, create the app user (if not `root`) and grant rights, then `db.create_all()`. Prints `DB_INIT_OK` / `DB_INIT_FAILED` and exits. Used by the installer. |
| `POS.exe --check-db` | Exit `0` if the DB is reachable with the current `.env`, else `1`. Lets the installer detect an already-working install on a re-run (so it doesn't ask for or overwrite credentials). |

Database creation lives inside the compiled exe so the installer needs nothing
but this one binary to provision everything.

## 5. Configuration

[config.py](config.py) loads a `.env` from `app_dir()` — the folder containing
`POS.exe` when frozen, or the source folder otherwise — with `override=True` so
installer-written values win over any inherited environment.

| Env var | Default | Purpose |
|---------|---------|---------|
| `SECRET_KEY` | `change-me-in-production` | Flask session signing key. |
| `DB_USER` | `root` | App's runtime DB user. |
| `DB_PASSWORD` | _(empty)_ | DB password (plain text in `.env`; see §13). |
| `DB_HOST` | `127.0.0.1` | DB host. |
| `DB_PORT` | `3306` | DB port. |
| `DB_NAME` | `pos_db` | Database name. |
| `CURRENCY` | `KES` | Display currency code. |
| `STORE_NAME` | `My Shop` | Shown in the window title and receipts. |
| `TAX_RATE` | `0` | Decimal rate, e.g. `0.16` for 16% VAT. |
| `UPDATE_URL` | _(empty)_ | Update manifest URL (read directly from `.env` by the updater). |
| `POS_HOST` | `127.0.0.1` | Address the local server binds to. |
| `POS_PORT` | `5000` | Preferred port. |

`server_uri()` returns a connection URI **without** a database name — used to
`CREATE DATABASE` on first run. `resource_path()` resolves bundled
`templates/` and `static/` both from source and from inside a PyInstaller
bundle (`sys._MEIPASS`).

## 6. Data model

Defined in [models.py](models.py). All money columns are `Numeric(12,2)`;
timestamps are UTC (`datetime.utcnow`).

```
users 1───* shifts 1───* sales *───1 users
                              │
                              *
                          sale_items *───1 products *───* stock_movements
```

| Table | Notable columns | Notes |
|-------|-----------------|-------|
| `users` | `username` (unique), `name`, `password_hash`, `role` (`admin`\|`cashier`), `is_active`, `must_change_password` | `set_password`/`check_password` via Werkzeug; `is_admin` property; `name` is the receipt display name. |
| `shifts` | `user_id`, `opened_at`, `closed_at` (NULL = open), `opening_float`, `closing_cash`, `note` | `summary()` returns the Z-report: per-method totals, expected cash, and over/short. |
| `products` | `barcode` (unique), `name`, `description`, `price`, `cost_price`, `quantity`, `reorder_level`, `is_active` | `low_stock` property = `quantity <= reorder_level`. Delete is a **soft delete** (`is_active=False`). |
| `sales` | `sale_number` (unique), `subtotal`, `tax`, `total`, `payment_method`, `amount_tendered`, `change_due`, `cashier`, `user_id`, `shift_id` | `cashier` is a name snapshot for receipts; `items` cascade-delete. |
| `sale_items` | `sale_id`, `product_id`, `barcode`, `name`, `unit_price`, `quantity`, `line_total` | Line snapshots so receipts survive later product edits. |
| `stock_movements` | `product_id`, `change_qty` (+in / −out), `movement_type` (`receive`\|`sale`\|`adjust`), `reference`, `note` | Append-only audit trail of every stock change. |

## 7. Authentication & request lifecycle

All defined in [app.py](app.py).

- **`load_current_user` (`before_request`)** resolves `g.user` from the session
  and `g.shift` (their open shift) once per request. A deactivated user is
  logged out immediately.
- **`inject_settings` (context processor)** exposes `STORE_NAME`, `CURRENCY`,
  `TAX_RATE`, `current_user`, and `current_shift` to every template.
- **`enforce_password_change` (`before_request`)** redirects a user flagged
  `must_change_password` to `/change-password` for every page except the change
  form, logout, and static files (APIs get a 403 JSON).
- **`@login_required`** — any signed-in user; APIs get a 401 JSON, pages
  redirect to `/login?next=…`.
- **`@admin_required`** — admins only; non-admins get a 403.

Sessions are Flask's signed cookies. The `next` redirect target is restricted to
same-site relative paths.

### JSON APIs

| Route | Method | Access | Purpose |
|-------|--------|--------|---------|
| `/api/product/<barcode>` | GET | auth | Look up one product by barcode. |
| `/api/products/search?q=` | GET | auth | Up to 25 active products by name/barcode. |
| `/api/products/grid` | GET | auth | All active products for the POS click grid. |

`404` → `404.html`, `403` → `403.html`, and `OperationalError` /
`InterfaceError` → `db_error.html` (HTTP 503), which also appends the traceback
to `pos_error.log` in the app folder.

## 9. Key flows

### Checkout (`POST /api/checkout`)

1. Reject if there is **no open shift** (409 `no_open_shift`) — every sale must
   attribute to a shift.
2. Re-resolve every cart line server-side against the live product row; reject
   unknown/inactive products, non-positive quantities, or insufficient stock.
   Prices and the cashier name come from the server, never the client.
3. Compute `subtotal`, `tax = subtotal * TAX_RATE` (rounded to cents), `total`.
4. For cash, require `amount_tendered >= total` and compute `change_due`;
   card / mobile money settle the exact amount.
5. In one transaction: insert the `Sale`, its `SaleItem`s, decrement each
   product's `quantity`, and log a `sale` `StockMovement` per line. The sale
   number is `SALE-YYYYMMDD-####` (sequential per day).

### Stock receive (`POST /api/stock/receive`)

Looks up the product by barcode; rejects a zero quantity and any write-off
larger than what's on hand (so the ledger always reconciles with
quantity-on-hand). Applies the delta and logs a `receive` (qty > 0) or `adjust`
(qty < 0) movement.

### Shifts

A cashier opens a shift with an opening float, rings up sales against it, then
closes it with the counted cash. `Shift.summary()` produces the Z-report:
sales count, totals by payment method, expected cash, and over/short.

## 10. Frontend

Server-rendered Jinja templates extend `base.html`. Two pages add interactivity:

- **[static/js/pos.js](static/js/pos.js)** — the POS screen. Loads the product
  grid, handles barcode-scan submits (a USB scanner types then presses Enter),
  live name/barcode filtering, the cart (add / change qty / remove, with stock
  guards), the payment modal (cash change preview), and posts to
  `/api/checkout`. All product HTML is escaped.
- **[static/js/stock.js](static/js/stock.js)** — the stock-receive form. Live
  product preview as you scan, then posts to `/api/stock/receive` and reloads.

Tax/currency are passed to the client via `window.POS_CONFIG`.

## 11. First-launch self-healing (`provision.py`)

`bootstrap()` runs before the window opens and returns `ok` / `handoff` /
`no_db`:

1. **Reachable** → ensure tables exist (`db.create_all()`), return `ok`.
2. **Service installed but stopped** → start it (elevating via UAC if needed),
   wait up to ~20 s, then ensure schema.
3. **Server up but DB/tables missing** → `CREATE DATABASE` + create tables.
4. **MariaDB not installed** → re-run the bundled installer
   (`XTPOS-Setup.exe` / `Repair-Setup.exe`, dropped next to the app at
   install time) and return `handoff` so the launcher exits while the installer
   takes over.

Everything is best-effort; on failure it returns `no_db` and the app shows the
friendly DB-error page.

## 12. Build, install, update, uninstall

### Build pipeline

`build-setup.bat` is the recommended one-command path. Everything is built
under `setup/`, with intermediates in `setup/build/` that are deleted at the
end — so the finished `setup/` folder contains **only** the shippable
`XTPOS-Setup.exe`. It:

1. `pyinstaller pos.spec` → a single `POS.exe` (**onefile** — `templates/`,
   `static/`, the icon, and pywebview's WebView2/.NET backend are folded inside
   the exe; there is **no** `_internal/` folder).
2. `pyinstaller uninstall.spec` → `Uninstall.exe` (single file).
3. `pyinstaller update.spec` → `Update.exe` (single file).
4. Stage the three exes **flat** into `setup/build/app/`.
5. `pyinstaller setup.spec` → `setup/XTPOS-Setup.exe`, which bundles
   `setup/build/app/` as `app_payload`, then deletes `setup/build/`.

Because all three executables are single files, the **installed** folder
(`C:\Program Files\XTPOS`) is flat: `POS.exe`, `Update.exe`, `Uninstall.exe`,
`.env`, `version.txt`, and the dropped repair copy of `XTPOS-Setup.exe` — no
nested directories.

`build.bat` builds only `POS.exe` (→ `dist/POS.exe`). `build-all.bat` +
[installer/pos.iss](installer/pos.iss) are an alternative Inno Setup path (needs
Inno Setup 6).

### Installer (`installer_app/setup_wizard.py`)

A tkinter wizard (Welcome → Settings → Install → Finish). `run_install()`:
installs MariaDB (downloads the MSI unless already present), installs the
WebView2 runtime, copies the app into `C:\Program Files\XTPOS`, drops a copy
of itself for self-repair, then — unless `--check-db` shows an already-working
install — initializes the database **before** writing `.env` (so a wrong
password never clobbers a working config). Finally it writes `version.txt`,
creates shortcuts, and registers an Add/Remove Programs entry.

`--selftest` prints environment detection + payload status as JSON (no install).

### Updater (`installer_app/update_wizard.py` → `Update.exe`)

Reads `UPDATE_URL` from the install folder's `.env`, fetches a JSON manifest
(`version`, `url`, `notes`, optional `sha256`), and if the manifest version
beats `version.txt` it downloads the zip, optionally verifies the digest, closes
the running POS, copies the new files over the install folder, and updates
`version.txt`. Files locked while running (e.g. `Update.exe` itself) are staged
as `*.new` and swapped in by a detached batch after the process exits (the batch
retries the move until it succeeds). Modes: GUI (default), `/silent`, `/check`.

**Publishing an update.** `build-setup.bat` runs [make_update.py](make_update.py)
after building, which writes `release/XTPOS-<version>.zip` (the app exes only —
never `.env` or `version.txt`) and `release/manifest.json` (with a sha256).
Workflow: bump the version (`bump_version.py`) → build → upload those two files
to your host. Set `UPDATE_BASE_URL` before building so the manifest's `url`
points at the right place, and set `UPDATE_URL` in each install's `.env` (or bake
a default into the installer) to the manifest URL. Installed apps then see the
new version via "Check for Updates". Test locally by serving `release/` with
`python -m http.server` and pointing `UPDATE_URL` at `http://127.0.0.1:8000/manifest.json`.

### Uninstaller (`installer_app/uninstall.py` → `Uninstall.exe`)

What the Windows "Apps & features" Uninstall button runs. Optionally drops
`pos_db` (on by default), optionally removes the MariaDB server (off by
default), removes shortcuts and the registry entry, and schedules deletion of
the install folder after it exits (since it lives inside that folder). Modes:
GUI (default), `/silent`, `/silent /purge-data`, `/silent /purge-data
/remove-mariadb`.

### `wizard_ui.py`

A minimal shared tkinter wizard framework (a `Wizard` driving `Page` subclasses,
Back/Next/Cancel, a header, and a `shared` dict) used by all three GUIs above.

## 13. Versioning & release

`VERSION` is the single source of truth. `bump_version.py [major|minor|patch|X.Y.Z]`
bumps it and promotes the `## [Unreleased]` section of [CHANGELOG.md](CHANGELOG.md)
to a dated release, leaving a fresh Unreleased section. `setup.spec` bundles
`VERSION` into the installer; at install time the wizard writes it to
`version.txt` in the install folder, which the updater compares against the
release manifest. (`build.bat` stamps `dist/version.txt` for the Inno path, and
the Inno Setup script reads `VERSION` directly.)

Typical release: edit code → fill in the Unreleased changelog → `python
bump_version.py minor` → `build-setup.bat` → publish the zip + manifest at
`UPDATE_URL`.

## 14. Running from source

```
copy .env.example .env          :: then set DB_PASSWORD (and STORE_NAME, etc.)
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python init_db.py               :: creates pos_db + tables + default admin
python seed.py                  :: optional: ~10 sample products
python launcher.py              :: opens the POS in its native window
```

`run.bat` automates all of the above. `app.py` can be run directly
(`python app.py`) as a raw dev server on port 5000, but the real app is always
launched via `launcher.py` / `POS.exe`.

To upgrade an older database in place (add users/shifts and the
`must_change_password` column), run `python migrate.py`.

## 15. Security notes

- **Local only.** The server binds to `127.0.0.1`; nothing is exposed to the
  network. The only outbound traffic is the one-time MariaDB/WebView2 download
  at install time and (if configured) update checks.
- **Passwords.** User login passwords are hashed (Werkzeug). The *database*
  password is stored in plain text in `.env` next to the exe — acceptable for a
  single-machine local POS. The MariaDB **root** password is never written to
  disk by the installer; it's passed to `POS.exe --init-db` via a temp JSON file
  that is deleted immediately after use.
- **Default credentials** (`admin` / `admin`) force a password change on first
  login via `must_change_password`. Change them before going live.
- **SQL.** Application queries go through SQLAlchemy. The few raw DDL statements
  (database/user creation) interpolate config values, not user input; passwords
  used in `CREATE USER` literals are single-quote-escaped.
